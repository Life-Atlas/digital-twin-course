What We Built

A LinkedIn Daily Brief app that gives you a morning dashboard of your LinkedIn activity: new connections, post analytics, engagement metrics, and AI-generated summaries — all in a single page you open with your coffee.

The key insight: instead of scrolling LinkedIn for 30 minutes, you get a 2-minute briefing of what matters. It pulls data from LinkedIn via MCP, processes it through Supabase Edge Functions, and presents it in a clean React dashboard.

Features at launch:

  • • Daily connection request summary with accept/ignore actions
  • • Post performance metrics (impressions, reactions, comments)
  • • Engagement feed with quick-reply drafts
  • • Authentication via Supabase with shared session across the LifeAtlas ecosystem
  • • Toast notification system for all actions
  • • Mobile-responsive dark theme matching LifeAtlas design system

Architecture

Browser (React + Vite)
    |
    +--> Supabase Auth (shared session)
    |
    +--> Supabase Edge Functions (Deno)
            |
            +--> /daily-brief      --> aggregates LinkedIn data
            +--> /connections      --> fetch + accept/ignore
            +--> /post-analytics   --> engagement metrics
            |
            +--> LinkedIn REST API (via stored OAuth tokens)

The frontend is a static React SPA deployed on Netlify. All LinkedIn API calls go through Supabase Edge Functions — never directly from the browser. This keeps tokens server-side and avoids CORS issues entirely.

Why Edge Functions? They run close to the user (Deno Deploy edge network), cold-start in under 100ms, and share the same Supabase auth context as the frontend. No separate backend to deploy.

Stack Choices and Why

LayerChoiceWhy
Frontend React + Vite + TypeScript Fast HMR, type safety, same stack as LifeAtlas core
Styling Tailwind CSS Consistent with LifeAtlas design system, no CSS files to manage
Validation Zod Runtime type checking for API responses — LinkedIn's API returns inconsistent shapes
Auth Supabase Auth Shared session across all LifeAtlas apps, JWT-based, zero custom code
Backend Supabase Edge Functions No server to manage, same auth context, edge-deployed globally
Hosting Netlify Git-push deploys, preview URLs, custom domain support

Zero lock-in. Every layer is replaceable. Vite can swap to Next.js. Supabase can swap to any Postgres + auth provider. Netlify can swap to Vercel or Cloudflare Pages. This is a LifeAtlas non-negotiable.

The Roast Methodology

We didn't plan this app in a Figma file for two weeks. We built it in rapid iterations using what we call the roast methodology: build fast, then have Claude Code brutally critique its own output, then fix everything it finds.

The cycle (repeat until score converges):

1

Build

Scaffold the feature as fast as possible. Don't polish. Get it working.

2

Roast

Ask Claude: "Roast this. Score it 1-10. Be brutal." It finds missing error handling, inconsistent patterns, accessibility gaps, security holes.

3

Fix

Address every item from the roast. Not just the easy ones — all of them.

4

Score

Re-roast. Compare scores. When you plateau at 8+/10, ship it.

This approach turns Claude from a code generator into a code reviewer. The adversarial loop catches things that a single-pass "generate and ship" workflow never would. We went from a 4/10 first draft to 8.5/10 in three roast cycles.

Key Patterns

Zod validation at the boundary

LinkedIn's API is loosely typed and returns different shapes depending on the endpoint version. We validate every response at the edge function boundary before it reaches the frontend. 

import { z } from "zod";

const ConnectionSchema = z.object({
  id: z.string(),
  firstName: z.string().default("Unknown"),
  lastName: z.string().default(""),
  headline: z.string().optional(),
  profileUrl: z.string().url().optional(),
  connectedAt: z.string().datetime().optional(),
});

// Parse with fallbacks — never crash on bad data
const connections = rawData
  .map(item => ConnectionSchema.safeParse(item))
  .filter(r => r.success)
  .map(r => r.data);

Shared auth across apps

All LifeAtlas apps share the same Supabase project. The Daily Brief app doesn't have its own login — it reads the existing session. If you're logged into LifeAtlas, you're logged into every satellite app. 

import { createClient } from "@supabase/supabase-js";

const supabase = createClient(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY
);

// Session persists across apps via shared Supabase project
const { data: { session } } = await supabase.auth.getSession();

REST endpoints with consistent error shape

Every Edge Function returns the same envelope. The frontend has one error handler, not ten. 

// Every response follows this shape
type ApiResponse<T> = {
  data: T | null;
  error: string | null;
  status: number;
};

// Frontend consumption
const { data, error } = await fetchBrief();
if (error) {
  toast.error(error);  // Consistent UX for all failures
  return;
}

Toast notifications for everything

Every user action gets feedback. Accept a connection? Toast. Error fetching data? Toast. No silent failures, no mystery loading states.

Lessons Learned

Start with the data, not the UI

We built the edge functions first, verified the data shapes, then built the frontend to match. Trying to design the UI before knowing what LinkedIn actually returns would have wasted hours.

The roast catches what tests miss

Tests verify correctness. Roasts verify quality. Missing loading states, inconsistent spacing, unclear error messages — these aren't bugs, but they're the difference between 6/10 and 9/10.

Zod is non-negotiable at API boundaries

LinkedIn's API returned null for fields documented as required. Without Zod's .default() and .optional(), the app would crash on real data that the test data didn't cover.

Ship the satellite, not the monolith

This is a standalone app that shares auth with the main platform. It ships independently, deploys independently, and can be killed independently. Every new feature doesn't have to be a PR to the monorepo.
